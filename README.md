## The Problem Every OT/ICS Security Team Faces

Industrial networks are not IT networks. The protocols are different, the risks are different, and the consequences of a breach — turbines tripping, pipelines stopping, substations going dark — are catastrophic. Yet most security teams are forced to use IT-centric tools that speak Ethernet but are blind to **Modbus**, **DNP3**, **S7comm**, and **EtherNet/IP**.

**OT-ICS-AI** was built to close that gap.

Drop in a ICS file captured from your OT network. Within minutes, receive a boardroom-ready security report — complete with threat findings, MITRE ATT&CK for ICS mapping, compliance gaps, and actionable recommendations — all generated by a hybrid engine that combines deterministic protocol rules with the reasoning power of large language models.

### 🧠 Hybrid Detection Engine: Rules + AI

OT-ICS-AI uses a **two-layer detection architecture**. This means critical anomalies are caught deterministically — without relying on any LLM — while the AI layer adds depth, context, and narrative that no rule set alone can provide.

```
┌──────────────────────────────────────────────────────────┐
│               LAYER 1 — RULE-BASED ENGINE                │
│  Runs on every packet. Zero false-negative on known      │
│  attack patterns. Works with NO LLM selected.            │
│                                                          │
│  Modbus Write Commands (FC5/6/15/16)     → HIGH          │
│  DNP3 Broadcast Address (0xFFFF)         → HIGH          │
│  DNP3 Multiple Masters / Same Outstation → CRITICAL      │
│  DNP3 Sequence Number Reset / Replay     → HIGH          │
│  DNP3 Repeated Control Bytes (DoS/Fuzz)  → MEDIUM        │
│  DNP3 Unsolicited Responses (FC=0x82)    → MEDIUM        │
│  Public IP → OT Protocol Access          → CRITICAL      │
│  Telnet / FTP Cleartext Sessions         → CRITICAL      │
│  Port Scan Detection (>20 ports)         → HIGH          │
│  Broadcast Storm (>50 packets)           → MEDIUM        │
└──────────────────────────────────────────────────────────┘
                          ↓
           Structured findings passed as context
                          ↓
┌──────────────────────────────────────────────────────────┐
│               LAYER 2 — LLM ANALYSIS                     │
│  Interprets, correlates, and narrates findings.          │
│  Optional — analysis runs fully without it.              │
│                                                          │
│  → Executive Summary (board-level language)              │
│  → MITRE ATT&CK for ICS technique mapping                │
│  → Risk Score (0–10) with full justification             │
│  → Compliance gap analysis (IEC 62443 / NERC CIP /       │
│    NIST SP 800-82)                                       │
│  → Prioritised remediation recommendations               │
└──────────────────────────────────────────────────────────┘
```

> **Key insight:** The LLM never sees raw packets. The rule engine extracts structured intelligence from bytes — function codes, sequence numbers, addressing anomalies — and the LLM reasons over that intelligence. This makes the analysis reliable even with smaller local models.

## Installation

### Requirements

```
Windows 10 / 11 (64-bit)
Linux x86_64
macOS 12+
No installation required — runs as a standalone executable
```

### Run — GUI Mode

Link: https://github.com/barrykfl-design/OT-ICS-AI-Security-Analyzer/blob/main/OTICS-AI.exe

Simply doenload and double-click **OTICS-AI.exe** (Windows).

The graphical interface launches immediately — no setup, no dependencies, no configuration files.

### Run — CLI Mode

```bash
# No LLM — instant rule-based analysis
OT-ICS-AI.exe capture.ICS --no-llm --format both

# With local Ollama
OT-ICS-AI.exe capture.ICS --llm ollama --model llama3

# With OpenAI
OT-ICS-AI.exe capture.ICS --llm openai --api-key sk-...

# With Anthropic Claude
OT-ICS-AI.exe capture.ICS --llm anthropic --api-key sk-ant-...

# Full options
OT-ICS-AI.exe capture.ICS \
  --llm anthropic \
  --api-key sk-ant-... \
  --format both \
  --output-dir ./reports \
  --verbose
```

## Enterprise Use Cases

### Power & Energy
Analyse DNP3 traffic between master stations and RTUs. Detect unauthorised polling, sequence anomalies, and rogue masters — the exact attack vectors used in the **Ukraine 2015/2016** grid attacks.

### Water & Wastewater
Monitor Modbus TCP commands to PLCs. Every write command is flagged and attributed — giving operators immediate visibility into who is changing setpoints and when.

### Oil & Gas
Identify unencrypted management protocols (Telnet, FTP) on control networks. Detect public-IP access to OT systems. Map lateral movement across process segments.

### Manufacturing
Analyse EtherNet/IP and S7comm traffic across factory floors. Build a complete device inventory from passive ICS analysis alone — no active scanning required.

### Critical Infrastructure Compliance
Generate compliance-mapped reports for **IEC 62443**, **NERC CIP**, and **NIST SP 800-82** audits in minutes, not weeks.

## Detection Rules Reference

All detection rules are deterministic, fully auditable, and documented:

| Rule | Detection | Severity |
|---|---|---|
| A | DNP3 application layer function code extraction | — |
| B | DNP3 sequence number tracking (bits 0–5 control byte) | — |
| C | DNP3 broadcast address usage (dst = 0xFFFF) | 🔴 HIGH |
| D | DNP3 unsolicited responses (FC = 0x82) | 🟡 MEDIUM |
| E | Multiple master IPs controlling same outstation | 🔴 CRITICAL |
| F | DNP3 sequence number reset / replay attack | 🔴 HIGH |
| G | Repeated identical control bytes — DoS / fuzzing | 🟡 MEDIUM |
| H | Broadcast frame storm detection | 🔴 HIGH |
| I | Multi-master outstation anomaly | 🔴 CRITICAL |
| J | Unsolicited response pattern analysis | 🟡 MEDIUM |
